### 人工审批组件
```
人工审批流程是怎么实现的呢？
在根Agent运行时，如果检查到用户的prompt中有“成本”“支付”这两个关键词时，会触发人工审批提醒。我们通过createApprove函数，在函数内部创建一个promise，和其对应的id，将其记录在Map中。向前端发送id和promise的信息。然后返回一个id和promise对象。然后根agent流程挂起，等待这个promise实例的完成。在前端，收到消息，并将消息更新到会话的内容列表中，页面渲染，展示人工审批的卡片，用户可以选择点击同意或拒绝。这两个按钮使用同一个函数，携带不同的审批信息，向后端的hil接口发送请求。接口看到如果请求体中的数据携带了id和原因，那就使用resolveApproval，完成我们在createApproval时创建的那个promise实例。流程等待结束，继续后续的执行。
```
### 为什么这个项目需要人工审核？你们是出于什么考虑加这个能力的？
```
这个项目里加入人工审核，核心是为了在高风险、高成本或者不可逆的动作前，给用户一个最终确认点，避免 Agent 直接执行带来错误后果。
因为 Agent 虽然能自动规划和调用工具，但在某些场景下，模型判断并不一定百分百可靠，如果完全自动执行，一旦误判，用户损失会比较大。

所以我们会把人工审核作为工作流里的一个安全阀。触发条件可以是一些高风险语义特征，比如涉及支付、成本、关键资源操作，或者后端把某些动作标记为需要确认。前端收到这类消息后，不只是展示内容，而是要把它转成一个明确的确认交互，让用户决定继续还是终止。这样系统既保留了自动化能力，也保留了风险可控性。”
```

### 报告附件组件

### 为什么要设计统一消息结构？如果没有这层抽象，会有什么问题？
```
“这个项目里我设计统一消息结构，核心原因不是单纯为了渲染方便，而是因为这个场景下后端返回的内容不是单一文本，而是一个持续流动的工作流事件流。里面既有普通文本，也有规划阶段、工具调用、工具返回、人工审核、最终报告这类不同语义的数据。

如果没有统一消息结构，前端就只能拿到一段段零散内容，再根据字段临时判断怎么渲染。这样会有几个问题：
第一，前端会充满大量 if else，消息类型一多，渲染逻辑和业务逻辑会耦合得很重；
第二，不同阶段的数据格式如果不稳定，前后端联调成本会很高，后端一改字段，前端很多地方都要跟着改；
第三，这个项目要做的是可视化工作流，而不是普通聊天，所以前端不仅要显示文本，还要知道当前处于哪个阶段、是否调用了工具、工具结果是什么、是否需要人工介入。没有统一协议，这些状态很难稳定地驱动 UI。”

“所以我会把消息抽象成统一结构，至少包含消息类型、角色、基础元信息，以及各类型自己的扩展字段。这样前端可以先按类型做分发，再由不同卡片组件处理各自的展示和交互。后面如果新增一种消息类型，扩展成本也比较低。”
```
### 这个项目里的消息大概分成哪些类型？每一类承担什么作用？
```
我会把消息分成几类。第一类是内容承载型消息，比如文本和附件，主要负责最终内容展示；第二类是流程可视化消息，比如子 Agent、工具调用、工具结果，用来把中间工作流展示出来；第三类是交互控制型消息，比如人工审核，它不只是展示，还会驱动前端产生确认/拒绝操作，并把结果回传后端，形成闭环。

之所以这样拆，是因为这个项目不是单纯聊天，而是一个有工作流、工具链和人工介入节点的系统。不同消息类型对应的是不同 UI 组件和不同状态流转，而不是简单把所有内容都当成文本渲染。”
```
