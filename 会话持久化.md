
### 为什么要持久化
```
这个项目需要历史会话管理和持久化，首先是因为用户的研究过程不是一次性消费内容，而是一个需要反复回看、对比和复盘的过程。如果刷新页面或者切换后内容丢失，用户体验会很差，尤其是长链路研究任务，前面的上下文和中间结论会直接失效。

另外，这个项目和普通聊天不太一样。我们保存的不只是几段对话文本，还包括工作流中的中间节点，比如工具调用结果、子 Agent 阶段信息、人工审核记录、最终报告附件等。这些内容本身就是研究过程的一部分，所以会话持久化不仅是为了‘保留聊天记录’，更是为了保留一次完整的研究过程，方便用户后续回顾、继续讨论或者下载结果。”
```
### 你为什么会先选择 localStorage 这类本地持久化，而不是直接把会话都存在服务端？这个取舍你怎么考虑的？
```
当时优先选择本地持久化，主要是基于项目阶段和成本考虑。这个项目在早期验证阶段，更重要的是先把会话管理、流式输出和工作流可视化这条主链路跑通。如果一开始就上服务端持久化，就要同时引入用户体系、会话表设计、接口设计、同步逻辑这些额外复杂度，开发成本会明显上升。

而 localStorage 的优点是实现简单、接入快，前端可以独立把会话的新建、切换、回放先做出来，所以对 demo 或原型阶段是比较合适的。

但它的局限也很明显。首先是容量和性能问题，会话越来越长、消息越来越多时，序列化和反序列化成本会上升；其次是数据只存在当前浏览器，无法多端同步，也不适合登录态用户长期保存；再就是安全性和稳定性都有限，用户清缓存或者更换设备后数据就没了。

所以我的理解是，本地持久化更像一个阶段性方案，适合快速验证交互和架构；如果进入生产环境，会优先迁移到服务端持久化，再结合本地缓存做体验优化。”
```
