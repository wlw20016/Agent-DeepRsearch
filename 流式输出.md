### 是怎么通过SSE实现流式输出的？
```
答：在初始化SSE时，会传入服务器的res对象，在res上配置响应头的相关内容，如Content-type为text/stream， Connect为keep-alive，这样就可以保证连接保活，并且浏览器可以持续接收流式数据。SSE的作用就是定义通信数据的结构{event, data},作为前后端通信的桥梁。
在llm层面，采用模拟大模型流式输出的形式。首先获取大模型的输出内容，在Agent处理时将大模型输出的内容进行分段处理，分为一段一段的chunk。接着再返回给前端。前端接收到响应内容后，会将返回的内容拼接到会话的消息内容中，数据更新引发视图的更新。在渲染新的消息时，会通过游标的形式，递归进行游标的更新，使其每次都加2，并且使用游标分割消息中的内容并显示在页面上。这样用户看到的就是文字不断地冒出的打字机效果。
```
### SSE和WebSockt的选择
```
答：DeepRcearch的场景是服务器持续向客户端发送消息的场景，所以需要SSE单向输出的特性而不是WebSockt这样的全双工通信。而且SSE更为轻量，且使用的是http协议，更好的兼容浏览器。
```
以下是从协议特性到项目实战场景的深度对比分析：
### 1. 核心区别对比
| 特性 | SSE (Server-Sent Events) | WebSocket |
| :--- | :--- | :--- |
| **通信方向** | **单向**：服务器向客户端推送数据 | **双向**：全双工通信，双方可同时收发 |
| **底层协议** | 标准 HTTP 协议 | 独立的 WebSocket 协议（需从 HTTP 升级） |
| **数据格式** | 文本流（默认 UTF-8） | 二进制或文本 |
| **连接保持** | 轻量级 HTTP 长连接 | 较重的持久连接 |
| **自动重连** | **原生支持**：浏览器会自动尝试重连 | 需开发者手动实现重连逻辑 |
| **代理/防火墙** | 兼容性极佳（就是普通 HTTP） | 可能会被某些严格的防火墙拦截 |

### 2. 为什么本项目选择 SSE 而不是 WebSocket？
在 `Agent-DeepRsearch` 这种 AI 对话/深度搜索场景下，选择 SSE 是基于**“架构契合度”**和**“工程性价比”**的深思熟虑：
#### A. 业务逻辑的单向性
AI 大模型的流式输出本质上是一个“**一问多答**”的过程：用户发送一个 Prompt（请求），大模型持续地吐出 Token（响应）。
* **场景匹配**：这种“请求-持续响应”的模型与 HTTP 的请求/响应范式高度契合。
* **避免过度设计**：WebSocket 的双向通信能力在这个场景下是冗余的。我们不需要在模型吐字的过程中，客户端还高频地向服务器发送数据。

#### B. 更好的流式中断管控 (`AbortController`)
在你的项目中，我们反复探讨了如何通过 `AbortController` 物理切断连接以停止大模型计费。
* **SSE 优势**：由于 SSE 基于标准 HTTP，我们可以直接利用浏览器成熟的 `fetch` API 和 `AbortSignal` 来精准控制请求的生命周期。
* **WebSocket 劣势**：要中断一个 WebSocket 中的特定“流”通常比较复杂，往往需要自定义一套信令协议来告诉后端“停止那个 ID 的任务”，增加了系统的复杂度和出错概率。

#### C. 原生重连与低开销
* **弹性架构**：SSE 内置了 `retry` 机制。如果网络出现瞬时波动，浏览器会自动携带 `Last-Event-ID` 尝试恢复连接，这在长耗时的 AI 搜索任务中非常实用。

#### D. 开发效率与兼容性
* **前端集成**：使用 `fetch-event-source` 库可以像发送普通 POST 请求一样处理 SSE。
* **后端实现**：在 Express 中，实现 SSE 只需要设置几个 Response Header 即可，而不需要引入专门的 WebSocket 库（如 `ws` 或 `socket.io`），保持了后端架构的简洁。

### 3. 面试总结话术
“在 `Agent-DeepRsearch` 项目中，我们选择了 SSE。因为我们的核心需求是**服务端向客户端的高频 Token 推送**，这与 SSE 的单向流特性完美契合。相比 WebSocket，SSE 更加轻量，且能原生复用现有的 HTTP/2 多路复用能力和 `AbortController` 中断机制。这不仅降低了服务器维护长连接的成本，还通过物理层面的请求阻断，有效地控制了模型 API 的计费成本。”
### 如何断开大模型的连接，以免页面关闭大模型还在产生token
```
答：使用AbortController。在前端，我们实现了一个useAgent Hook函数，该函数返回了当前连接状态，开始agent函数，结束agent函数和暂停函数。当用户输入完内容，点击发送按钮以后，会触发更新数据函数的回调，该回调内部会获取用户的输入并作为prompt，触发开始agent函数，在开始agent函数内部，首先会实例化AbortController并且更新相关的useRef状态来保存这个对象，再通过fetch向后端发送请求。如果用户关闭了页面，触发组件卸载，就会触发开始agent函数中的副作用函数，调用关闭agent函数，并且调用abort方法来阻断浏览器和服务器的fetch连接。
在后端服务器，创建SSE的时候会实例化AbortController，并且通过res.onclose监听关闭事件，一但连接断开，就触发回调，使用abort方法断开服务器与大模型的连接。因为我们调用大模型的方法，实际上是一个启动http请求的工具类，我们把sse中的abort对象放了进去，sse发生了熔断，发起的客户端到llm的请求也会中断。
```
